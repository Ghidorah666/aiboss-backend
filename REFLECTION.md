## 1. 代码审查方法论

### 1.1 对比式学习

**方法**:先读旧代码,再读新代码,对比差异,理解设计意图。

**本次实践**:
- 旧代码:routes/payment.js 用 `require('alipay-sdk')` 做查询
- 新代码:用原生 `fetch` 替代
- 理解:减少依赖,简化部署,但保留SDK用于复杂操作(pageExecute)

**适用场景**:接手别人的项目、大版本升级、代码审查。

---

### 1.2 遗漏功能识别

**方法**:对比新旧代码,找出被删除但仍有价值的功能。

**本次实践**:
- 旧代码有 `confirm` 端点(发布者确认完成+打款)
- 新代码删除了这个端点
- 判断:这是核心业务闭环,删除会导致接单者无法获得报酬

**教训**:代码简化时要区分"可以删"和"必须保留"的功能。

## 2. 架构决策记录

### 2.1 密钥文件优先策略

**决策**:读取支付宝密钥时,文件优先于环境变量。

**原因**:
- 环境变量容易配错(把应用公钥当支付宝公钥)
- 文件可以从支付宝后台直接下载,不易混淆
- 文件在.gitignore中,不会泄露

**启示**:敏感配置用文件管理比环境变量更安全可靠。

---

### 2.3 任务闭环设计（PRD v1.0 完整实现）

**PRD要求的任务状态机**：
```
pending(待支付) → active(可接单) → accepted(已接单) → submitted(已提交证明)
    → reviewing(审核中) → completed(已完成+打款)
                        → rejected(审核拒绝，可重新提交)
```

**实现的核心流程**：
1. 发布任务 → status=pending
2. 用户支付 → settlePaidOrder → status=active
3. 工作者接单 → recordOps.accept → accepted_count+1
4. 工作者提交证明 → status=submitted
5. 发布者审核通过 → recordOps.completed + taskOps.completed + 给工作者加余额
6. 发布者审核拒绝 → recordOps.rejected + taskOps回退到accepted

**提现流程**：
1. 用户申请提现 → 冻结余额 + 创建withdrawal记录
2. 管理员审核通过 → 解冻（扣冻结）
3. 人工打款 → 标记paid
4. 管理员拒绝 → 退还余额 + 解冻

**教训**：每新增一个业务流程，先画状态机图，再写代码。状态机不完整，代码一定有断点。

### 2.4 幂等性设计

**决策**:`settlePaidOrder()` 函数同时被notify和settle调用。

**设计**:
- 检查订单状态,已paid返回alreadyPaid
- 未paid才执行更新
- 无论谁调用,结果一致

**启示**:支付相关操作必须幂等,否则会出现重复加余额等问题。

---

### 2.3 人工审核提现

**决策**:提现不直接调支付宝转账,改为人工审核。

**原因**:
- 自动转账风险高(错付、重复付款不可逆)
- 人工审核可以增加安全检查
- 202状态码表示"已受理,待审核"

**启示**:涉及资金流出的操作,宁可慢不可错。

## 3. 项目管理经验

### 3.1 MD文档体系的价值

**观察**:SPEC.md、KNOWLEDGE.md、REFLECTION.md、BUGLIST.md 四个文档覆盖了项目的不同维度。

**价值**:
- SPEC.md:新人接手时快速了解项目全貌
- KNOWLEDGE.md:避免重复踩坑
- REFLECTION.md:沉淀方法论,提升决策质量
- BUGLIST.md:追踪问题,不遗漏

**教训**:文档不是负担,是投资。写文档的时间 < 反复踩坑的时间。

---

### 3.2 代码交接的完整性

**观察**:PROJECT_HANDOVER.md + PROJECT_HANDOVER_UPDATE.md 提供了完整的项目交接。

**要素**:
- 账号信息(服务器、GitHub、支付宝)
- 环境变量清单
- 密钥文件清单
- 当前状态(已修复BUG、待优化项)
- 关键路由表

**启示**:交接文档要包含"另一个AI能直接接手"的全部信息。

## 4. 下一步行动原则

### 4.1 闭环优先

**原则**：每个功能必须有完整的闭环，不能有断点。

**已完成闭环**：
- ✅ 充值闭环：创建订单 → 支付 → 补偿查询 → 入账
- ✅ 任务闭环：发布 → 支付激活 → 接单 → 提交 → 审核 → 打款
- ✅ 提现闭环：申请 → 冻结余额 → 管理员审核 → 打款/退还

**当前待补**：
- 站内通知（接单/完成/审核结果消息）
- 管理后台UI（目前只有API，无前端页面）
- AI Agent API Key 认证

---

### 4.2 安全优先

**原则**:涉及资金的操作,宁可多做检查,不能图省事。

**当前缺失**:

- JWT强密钥
- 提现审核流程

**行动**:在推进新功能前,先补齐安全措施。
