版本：v1.0  
更新时间：2026-05-05  
产品定位：AI 时代的人类劳动力调度与结算平台  
当前形态：Node.js + Express + sql.js + 前端静态页 + 支付宝支付/补偿结算  

---

## 1. 产品概述

AIBoss 是一个让 AI 和人类协作完成现实世界任务的平台。AI 或人类发布任务，人类工作者接单执行，平台通过支付宝完成充值、支付、提现和结算。

一句话定义：

> 让 AI 成为任务发起方，让人类成为现实世界执行者，让支付和交付形成闭环。

核心场景包括：线下签到、拍照取证、信息采集、代办执行、问卷调研、跑腿协作、AI Agent 自动发单等。

---

## 2. 背景与机会

大模型已经能规划、沟通和决策，但无法直接完成大量现实世界动作，例如到店拍照、现场确认、人工操作、线下代办、物理世界信息采集。AIBoss 的机会在于把 AI 的任务规划能力与人类的物理执行能力连接起来。

市场痛点：

- AI 有大量现实世界执行需求，但缺少稳定的人类执行网络。
- 传统众包平台偏人工发单，不适合 AI Agent 高频、自动、结构化调用。
- 小额任务支付链路复杂，发单、接单、结算、对账成本高。
- 任务完成过程缺少标准化状态、证据提交、回调通知与交易记录。
- 对个人工作者而言，碎片化任务入口分散，结算可信度不稳定。

产品机会：

- 把 AIBoss 打造成 “AI 的人类执行层”。
- 先从人民币支付宝闭环切入，降低真实交易落地难度。
- 后续向 API、Skill、Agent 工具市场、跨境 USDC 结算扩展。

## 3. 产品目标

- 完成支付宝充值与任务支付链路。
- 完成支付回调验签与补偿查询，保证支付状态最终一致。
- 完成个人中心、钱包、交易记录、移动端基础适配。
- 完成数据库备份与最小运维能力。

- 增加提现审核后台。

v2.0 阶段目标：

- 支持多城市、多品类任务供给。
- 建立工作者等级、信用和质量评分体系。
- 支持企业/AI Agent 批量发单。
- 支持 USDC 或其他跨境结算方式。
- 打造面向 AI Agent 的标准化现实世界执行 API。

## 4. 用户角色

### 4.1 人类工作者

目标：

- 找到简单、可信、可快速结算的任务。
- 完成任务后获得人民币收益。
- 能查看自己的任务、钱包、交易记录。

关注点：

- 任务是否真实。
- 报酬是否明确。
- 完成标准是否清楚。
- 收款是否及时。
- 平台是否会无故扣款或拒付。

### 4.2 人类发单者

目标：

- 发布现实世界任务。
- 支付任务预算。
- 查看任务状态。
- 获取工作者提交的证明。

关注点：

- 任务能否被快速接单。
- 任务质量是否可控。
- 资金是否安全。
- 未完成任务如何退款或重发。

### 4.3 AI Agent 发单者

目标：

- 通过 API 自动创建任务。
- 自动支付或使用预充值余额。
- 通过 callback_url 接收任务状态变化。
- 用结构化数据获取人类执行结果。

关注点：

- API 是否稳定。
- 状态机是否明确。
- 回调是否可信。
- 任务结果是否可被机器解析。

### 4.4 平台运营/管理员

目标：

- 管理用户、任务、订单、提现和投诉。
- 处理支付异常、人工审核和风控。
- 监控平台健康状态。

关注点：

- 是否有完整交易记录。
- 是否能追踪订单状态。
- 是否能处理争议。
- 是否有备份和恢复能力。

## 5. 核心用户故事

### 5.1 工作者接单

作为工作者，我希望浏览任务广场，看到报酬、地点、任务要求和状态，以便判断是否接单。

验收标准：

- 可查看任务列表。
- 可按任务类型筛选。
- 登录后可接单。
- 已接任务出现在个人中心。
- 不允许重复接同一任务。

### 5.2 发单者发布任务

作为发单者，我希望填写标题、描述、类型、地点、报酬并发布任务，以便寻找人类执行者。

验收标准：

- 未登录用户发布时跳转登录。
- 标题、类型、报酬必填。
- 报酬必须大于 0。
- 创建后任务为 pending。
- 支付成功后任务变为 active。

### 5.3 用户充值

作为用户，我希望在个人中心发起充值，通过支付宝支付后余额自动增加。

验收标准：

- 可输入充值金额。
- 创建订单后返回支付表单。
- 支付成功后订单变为 paid。
- 充值订单只入账一次。
- 支付宝回调失败时，可通过补偿查询完成入账。

### 5.4 查看交易记录

作为用户，我希望看到所有充值和支付订单，方便确认资金状态。

验收标准：

- 展示订单号、金额、币种、状态、创建时间。
- pending 订单显示补偿查询按钮。
- paid 订单不重复执行结算。
- 移动端可横向滚动或良好展示。

### 5.5 AI Agent 自动发单

作为 AI Agent，我希望通过 API 创建任务并设置 callback_url，以便任务完成后自动接收状态通知。

验收标准：

- API 支持创建任务。
- 可传入 callback_url。
- 任务支付成功后激活。
- 后续任务完成、审核、放款状态可回调。
- 回调需要签名，防止伪造。

## 7. 核心业务流程

### 7.1 充值流程

1. 用户进入个人中心。
2. 输入充值金额。
3. 前端调用 `/api/payment/qrcode`。
4. 后端创建 pending 订单。
5. 后端生成支付宝支付表单。
6. 用户跳转支付宝完成支付。
7. 支付宝调用 `/api/payment/notify`。
8. 后端验签。
9. 后端查询订单。
10. 如果未支付过，订单状态改为 paid。
11. 用户人民币余额增加。
12. 用户回到 dashboard。
13. 若 notify 未到达，前端调用 `/api/payment/settle/:orderNo` 补偿查询。

### 7.2 发布任务流程

1. 用户登录。
2. 填写任务信息。
3. 后端创建 pending 任务。
4. 用户支付任务金额和服务费。
5. 支付成功后任务状态变为 active。
6. 任务出现在任务广场。
7. 工作者接单。
8. 工作者提交完成证明。
9. 发单者审核。
10. 审核通过后工作者余额增加。
11. 任务状态变为 completed。

### 7.3 提现流程

1. 工作者在个人中心申请提现。
2. 输入提现金额和支付宝账号。
3. 系统校验余额。
4. 创建提现申请 pending_review。
5. 管理员审核。
6. 人工打款或调用转账接口。
7. 提现状态变为 paid。
8. 用户余额扣减或冻结释放。

## 8. 状态机设计

### 8.1 任务状态

- pending：待支付/待激活
- active：可接单
- accepted：已被接单
- submitted：工作者已提交完成证明
- reviewing：发单者审核中
- completed：已完成并结算
- rejected：审核拒绝
- cancelled：已取消
- expired：已过期

### 8.2 订单状态

- pending：待支付
- paid：已支付
- failed：支付失败
- closed：订单关闭
- refunded：已退款

### 8.3 提现状态

- pending_review：待审核
- approved：审核通过
- rejected：审核拒绝
- paid：已打款
- failed：打款失败

### 8.4 接单记录状态

- accepted：已接单
- submitted：已提交证明
- completed：已完成
- rejected：提交被拒绝
- abandoned：工作者放弃

## 12. 支付与结算要求

### 12.1 支付安全

- notify 默认必须验签。
- 不允许生产环境跳过验签。
- 支付宝公钥必须与应用配置匹配。
- 订单结算必须幂等。
- 金额必须以后端订单金额为准。
- 不信任前端传入的支付状态。

### 12.2 补偿机制

适用情况：

- 支付宝 notify 未到达。
- 用户支付后直接返回 dashboard。
- 服务器临时异常导致回调失败。

机制：

- 前端携带 orderNo 或 out_trade_no 自动调用 settle。
- pending 订单允许用户手动点击补偿查询。
- 后端主动查询支付宝交易状态。
- 支付成功则走统一结算函数。

### 12.3 对账

后续建议：

- 每日定时扫描 pending 订单。
- 对超过 5 分钟的 pending 订单主动查询。
- 对 paid 订单记录 trade_no。
- 后台支持按订单号查询。

## 13. 风控与反作弊

MVP 风控：

- 同一用户不可重复接同一任务。
- 未支付任务不可出现在任务广场。
- 金额必须大于 0。
- 订单不可重复结算。
- 提现必须人工审核。

v1.1 风控：

- 限制新用户每日接单数。
- 同 IP/设备异常注册检测。
- 高频发布任务限制。
- 工作者提交证明需要时间、位置或图片。
- 投诉率过高用户降权。

v2.0 风控：

- 人脸/实名/KYC 可选。
- 地理位置校验。
- 图片 EXIF 校验。
- AI 识别重复图片和伪造图片。
- 信用分系统。

## 14. 非功能需求

### 14.1 性能

- API P95 响应时间小于 500ms。
- 静态页面首屏加载小于 2s。
- 任务列表默认分页，避免一次返回过多数据。

### 14.2 可用性

- 支付链路最终一致。
- 数据库每日备份。
- 服务启动失败应明确报错。
- 健康检查 `/api/health` 可用。

### 14.3 安全

- JWT_SECRET 必须生产随机化。
- `.env` 不提交 GitHub。
- 数据库文件不提交 GitHub。
- 支付私钥不提交 GitHub。
- 支付 notify 必须验签。

### 14.4 移动端

- 主要页面适配 375px 宽度。
- 导航不遮挡内容。
- 交易记录可横向滚动或卡片化。
- 按钮高度适合触控。

## 19. 产品原则

- 资金安全优先于体验速度。
- 支付状态以后端和支付宝查询为准。
- 任何结算逻辑必须幂等。
- 任务要求要清楚，减少争议。
- 移动端优先，因为工作者大多在手机上操作。
- 先做小额、低风险、高频任务，再扩展复杂任务。
- 让 AI 更容易发单，让人更容易赚钱。
